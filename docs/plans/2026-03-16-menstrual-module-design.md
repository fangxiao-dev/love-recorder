# 微信小程序共享月经记录模块设计

**日期：** 2026-03-16

## 目标

构建一个用于微信小程序的恋爱记录产品 MVP。长期方向是“情侣细节记录空间”，但第一版只交付月经记录模块，并确保它既可单人使用，也可升级为共享协作。

## 产品定位

- 产品形态：微信小程序
- 产品方向：情侣细节记录空间
- MVP 范围：仅做月经记录模块
- 使用模式：单人可用，支持后续共享给伴侣共同查看和维护
- 气质：底层是实用工具，表达上保留少量关系感和温度

## 核心原则

1. 先可用，再共享。用户不需要先绑定情侣关系才能开始使用。
2. 共享的是同一份模块实例，不复制数据。
3. 默认交互追求低负担，先完成记录，再允许补充细节。
4. 首页先回答“当前状态”，不是先展示完整自然月日历。
5. MVP 只做高频核心动作，不引入复杂分析、社交互动或健康建议。

## 核心用户故事

### 场景 1：单人记录

用户自己使用月经记录模块，完成开始、结束、补录和查看历史等基本操作。

### 场景 2：共享出去

用户找到伴侣后，可创建或加入共享空间，并将自己的月经记录模块挂载到共享空间中。此后：

- 在个人入口中仍可看到该模块
- 模块状态显示为“已共享”
- 自己和伴侣进入后看到的是同一份数据

### 场景 3：撤回共享

用户可以将该模块从共享空间中解绑。解绑后：

- 对方失去访问权限
- owner 仍保留原始模块和全部历史数据
- 模块恢复为私有状态

## 信息架构

第一版建议采用以下页面结构：

1. 登录/进入页
2. 我的模块页
3. 共享空间入口页
4. 月经记录详情页
5. 记录编辑/补录页

说明：

- “我的模块”负责 owner 视角和归属管理
- “共享空间”负责协作视角
- 两个入口最终都指向同一份模块实例

## 数据模型

### User

代表登录用户。

### Module Instance

某个模块的一份具体实例，例如“我的月经记录”。它始终有明确 owner，记录数据跟随实例本身存在。

### Shared Space

一个可以邀请其他人加入的共享空间，用于承载被共享进来的模块实例。

### Access / Membership

描述谁在共享空间中，以及谁对某个模块实例拥有何种访问能力。

## 共享规则

### Owner 权限

owner 拥有完整控制权，可以：

- 编辑所有记录
- 共享模块
- 撤回共享
- 删除模块实例
- 修改模块基础设置

### 共享成员权限

共享成员在 MVP 中可以：

- 查看记录
- 新增普通记录
- 修改普通字段

共享成员不能执行 owner 级别操作，例如：

- 撤回共享
- 删除模块实例
- 修改关键归属设置

## 月经记录模块边界

### MVP 必做

1. 记录一次周期
2. 查看当前状态
3. 查看历史记录
4. 使用简单提醒

### 单条记录字段

- 开始日期
- 结束日期
- 流量等级
- 疼痛等级
- 备注

说明：流量、疼痛和备注为可选补充信息，不应阻塞快速记录。

### MVP 不做

- 症状标签体系
- 复杂统计图表
- AI 解读
- 备孕模式
- 排卵/体温体系
- 情绪日记流
- 重型互动功能

## 首页交互设计

月经记录首页应采用“状态卡 + 连续时间带”的结构，而不是传统自然月日历。

### 第一屏内容

1. 主状态
2. 次状态
3. 连续时间带
4. 快捷动作

### 主状态

只展示一个当前状态：

- 经期中：显示“经期中第 X 天”
- 非经期中：显示“距离上次开始第 X 天”

### 次状态

显示轻量预测信息：

- 预计下次还有 X 天
- 或预测窗口

### 可视化展示

建议使用最近 30 至 45 天的连续时间带，而不是自然月日历。

原因：

- 更符合身体周期，而不是自然月边界
- 避免跨月理解中断
- 空白日期更少，信息密度更高
- 更适合快速补录和查看最近状态

时间带中建议表达：

- 已记录经期区间
- 今天
- 预测窗口

## 快捷记录交互

MVP 记录交互原则：一步记录，细节后补。

### 主要动作

- 今天来了
- 今天结束了
- 补录一段
- 记录异常

### 交互规则

- 点击“今天来了”后，直接将今天记为开始日
- 点击“今天结束了”后，直接补充结束日
- 补录历史时，采用连续区间选择
- 默认不强制填写流量、疼痛、颜色等细节
- 只有特殊情况才建议补充异常说明

## 协作感设计

共享模式采用“中共享”强度，不做过重的社交表达。

### MVP 中保留的协作信息

- 模块已共享状态
- 最近一次是谁补充了记录

### MVP 中暂不做

- 复杂互动反馈
- 对方确认流程
- 情绪化社交组件
- 高强度关怀任务系统

## 与传统经期 App 的差异

相较于典型单人经期 App，本产品应有以下明显差异：

1. 先回答状态，再展示图形
2. 用连续时间带替代自然月日历
3. 弱化冗余字段采集，提升快速记录效率
4. 产品根节点是“模块可共享”，不是“单人健康管理”
5. 页面表达保持中性和克制，避免模板化“粉色经期工具”观感

## 微信小程序约束

设计和实现时需要明确以下前提：

- 页面层级和交互需适配微信小程序导航模型
- 主要操作必须能在单手、短时使用场景下快速完成
- 组件设计应适配小程序常见的卡片、弹层、底部操作区模式
- 视觉与动效需克制，避免过重 H5 化交互
- 未来若接入订阅消息或提醒能力，应从小程序能力边界出发设计

## 后续扩展方向

在不推翻当前结构的前提下，后续可增加：

- 纪念日模块
- 偏好记录模块
- 打卡模块
- 愿望清单模块
- 共享时间线入口

这些模块都应复用“模块实例 + 可挂载共享空间”的同一套产品模型。
