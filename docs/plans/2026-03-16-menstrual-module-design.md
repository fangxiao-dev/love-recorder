# 微信小程序共享月经记录模块设计

**日期：** 2026-03-16

## 目标

构建一个用于微信小程序的恋爱记录产品 MVP。长期方向是“情侣细节记录空间”，但第一版只交付月经记录模块，并确保它既可单人使用，也可升级为共享协作。

## 产品定位

- 产品形态：微信小程序
- 产品方向：情侣细节记录空间
- MVP 范围：仅做月经记录模块
- 使用模式：单人可用，支持后续共享给伴侣共同查看和维护
- 气质：底层是实用工具，表达上保留少量关系感和温度

## 核心原则

1. 先可用，再共享。用户不需要先绑定情侣关系才能开始使用。
2. 共享的是同一份模块实例，不复制数据。
3. 默认交互追求低负担，先完成记录，再允许补充细节。
4. 首页先回答“当前状态”，不是先展示完整自然月日历。
5. MVP 只做高频核心动作，不引入复杂分析、社交互动或健康建议。

## 核心用户故事

### 场景 1：单人记录

用户自己使用月经记录模块，完成开始、结束、补录和查看历史等基本操作。

### 场景 2：共享出去

用户找到伴侣后，可创建或加入共享空间，并将自己的月经记录模块挂载到共享空间中。此后：

- 在个人入口中仍可看到该模块
- 模块状态显示为“已共享”
- 自己和伴侣进入后看到的是同一份数据

### 场景 3：撤回共享

用户可以将该模块从共享空间中解绑。解绑后：

- 对方失去访问权限
- owner 仍保留原始模块和全部历史数据
- 模块恢复为私有状态

## 信息架构

第一版建议采用以下页面结构：

1. 登录/进入页
2. 我的模块页
3. 共享空间入口页
4. 月经记录详情页
5. 记录编辑/补录页

说明：

- “我的模块”负责 owner 视角和归属管理
- “共享空间”负责协作视角
- 两个入口最终都指向同一份模块实例

## 数据模型

### User

代表登录用户。

### Module Instance

某个模块的一份具体实例，例如“我的月经记录”。它始终有明确 owner，记录数据跟随实例本身存在。

### Shared Space

一个可以邀请其他人加入的共享空间，用于承载被共享进来的模块实例。

### Access / Membership

描述谁在共享空间中，以及谁对某个模块实例拥有何种访问能力。

## 共享规则

### Owner 权限

owner 拥有完整控制权，可以：

- 编辑所有记录
- 共享模块
- 撤回共享
- 删除模块实例
- 修改模块基础设置

### 共享成员权限

共享成员在 MVP 中可以：

- 查看记录
- 新增普通记录
- 修改普通字段

共享成员不能执行 owner 级别操作，例如：

- 撤回共享
- 删除模块实例
- 修改关键归属设置

## 月经记录模块边界

### MVP 必做

1. 记录一次周期
2. 查看当前状态
3. 查看历史记录
4. 使用简单提醒

### 单条记录字段

- 开始日期
- 结束日期
- 流量等级
- 疼痛等级
- 备注

说明：流量、疼痛和备注为可选补充信息，不应阻塞快速记录。

### MVP 不做

- 症状标签体系
- 复杂统计图表
- AI 解读
- 备孕模式
- 排卵/体温体系
- 情绪日记流
- 重型互动功能

## 首页交互设计

月经记录首页应采用“状态卡 + 周期日历 + 原地编辑面板”的结构。默认日历是 **Cycle Window（三周窗口）**，Month View 作为辅助浏览模式存在，但编辑动作始终在首页内联完成，而不是把用户频繁带去独立页面。

### 第一屏内容

1. 主状态
2. 次状态
3. 周期日历
4. 日期详情 / 原地编辑面板
5. 快捷动作

### 主状态

只展示一个当前状态：

- 经期中：显示“经期中第 X 天”
- 非经期中：显示“距离上次开始第 X 天”

### 次状态

显示轻量预测信息：

- 预计下次还有 X 天
- 或预测窗口

### 日历展示

首页日历采用双模式：

- 默认：Cycle Window（三周、21 天、按 1 周步进）
- 辅助：Month View（整月浏览）

Cycle Window 的中心定位优先级为：

1. 当前经期
2. 预测经期
3. 今天

日历中需要表达：

- 已记录经期区间
- 预测窗口
- 今天
- 选中日期

Month View 的目的主要是回看和定位历史日期，不承担独立编辑页职责。

### 日期点击后的原地编辑

点击任意日期后，日历下方应直接显示该日期对应的状态面板，而不是弹出“无数据”提示，也不是默认跳转到独立详情页。

这个面板的职责是：

- 显示当前选中日期
- 告诉用户这一天当前被系统判断为什么状态
- 提供最少必要的修改动作

MVP 中，日期面板优先围绕“月经开始 / 月经结束 / 是否异常”组织，而不是铺开大量字段。

## 快捷记录交互

MVP 记录交互原则：一步记录，细节后补。

### 主要动作

- 今天来了
- 今天结束了
- 补录一段
- 记录异常

### 交互规则

- 点击“今天来了”后，直接把今天记为一个周期的开始日
- 如果一个周期已经开始，在默认经期天数范围内，后续日期的面板主动作应切换为“月经走了：是/否”
- 当用户在某一天选择“月经走了：是”时，该天即成为该周期的结束日
- 如果用户没有手动结束，则系统在默认经期天数的最后一天自动闭环
- 默认经期天数初始值为 7 天，后续允许用户调整
- “补录一段”是单独的范围补录能力，不等于单日点击编辑
- 补录历史时，采用连续区间选择，并在首页内完成，不再跳去占位页
- 默认不强制填写流量、疼痛、颜色等细节
- 只有特殊情况才建议补充异常说明

### 单日状态规则

单日点击后的面板遵循以下规则：

1. 如果该日期当前没有任何状态，默认主操作展示为“月经来了”
2. 如果该日期落在某个进行中的默认经期窗口内，主操作展示为“月经走了：是/否”
3. “月经走了：是” 表示该选中日就是本次周期的最后一天
4. 日期是否属于经期，不只取决于用户是否逐日补录，还受“开始日 + 默认经期天数 + 手动结束日”共同影响

### 异常记录规则

异常记录需要从“详尽采集”改为“仅在偏离预期时才展开”。

MVP 的判断方式分两层：

1. 符合预期
   - 用户觉得这一天整体正常，系统保留默认状态
   - 用户不需要填写流量、颜色、疼痛等级
2. 异常状态
   - 只有当用户主动判断“不符合预期”时，才进入异常记录
   - 异常记录中才展开更细的流量、颜色、疼痛或备注项

这样做的目标是：

- 保持记录成本低
- 避免把“正常情况”也变成多字段表单
- 让异常信息保持高信号，而不是被日常微小波动稀释

## 协作感设计

共享模式采用“中共享”强度，不做过重的社交表达。

### MVP 中保留的协作信息

- 模块已共享状态
- 最近一次是谁补充了记录

### MVP 中暂不做

- 复杂互动反馈
- 对方确认流程
- 情绪化社交组件
- 高强度关怀任务系统

## 与传统经期 App 的差异

相较于典型单人经期 App，本产品应有以下明显差异：

1. 先回答状态，再展示图形
2. 用周期窗口作为默认视图，并允许整月浏览作为辅助模式
3. 单日点击直接在首页原地编辑，而不是强依赖详情页跳转
4. 弱化冗余字段采集，提升快速记录效率
5. 只有异常时才展开更细的记录项
4. 产品根节点是“模块可共享”，不是“单人健康管理”
6. 页面表达保持中性和克制，避免模板化“粉色经期工具”观感

## 微信小程序约束

设计和实现时需要明确以下前提：

- 页面层级和交互需适配微信小程序导航模型
- 主要操作必须能在单手、短时使用场景下快速完成
- 组件设计应适配小程序常见的卡片、弹层、底部操作区模式
- 视觉与动效需克制，避免过重 H5 化交互
- 未来若接入订阅消息或提醒能力，应从小程序能力边界出发设计

## 后续扩展方向

在不推翻当前结构的前提下，后续可增加：

- 纪念日模块
- 偏好记录模块
- 打卡模块
- 愿望清单模块
- 共享时间线入口

这些模块都应复用“模块实例 + 可挂载共享空间”的同一套产品模型。
